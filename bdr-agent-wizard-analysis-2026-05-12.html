The 9 Playbooks Analyzed

Each collapsible section below summarizes a real customer script that was analyzed to produce this document. These span D2C, SaaS, financial services, auto, and professional services verticals.

Outbound Warm 8. Dark Horse Scheduler (CPA Firm)

  • Goal: Confirm existing appointment + collect additional context

Part 1: Deep Analysis

For PM and Product Designers — patterns extracted from 9 real customer playbooks

1.1 Universal Call Flow Architecture

All 9 scripts share a common flow pattern that can be decomposed into 7 stages. Regardless of industry, call direction, or complexity, every playbook follows this progression:

1

Opening

Agent identifies self + company + purpose. Always uses the prospect's name. Sets the first impression and establishes legitimacy.

2

Verification

Confirm speaking to the right person. Sometimes includes recorded line disclosure (Honeycomb). May include identity confirmation (Apple DR).

3

Permission / Framing

"Is now a good time?" + set expectations for what the call will cover. Frames the call as helpful, not intrusive. Dark Horse asks "permission to ask a few questions."

4

Discovery / Qualification

Core questions. This is where scripts diverge most. Can be simple (3 questions for LavieLabs) or complex (multi-gate with branching for Amanda). The heart of every script.

5

Objection Handling

Predefined responses to common pushbacks. Some scripts have this as a separate section (UBS with 16 objections), others embed it inline within the flow.

6

Outcome Routing

Based on qualification: (a) hot transfer to a human rep, (b) schedule a callback or meeting, or (c) disqualify with a soft letdown. All scripts have at least two outcome paths.

7

Closing

Recap next steps, confirm understanding, thank the prospect. Hold/waiting behavior is defined separately in some scripts (LavieLabs, Warranty First).

Design Insight: The wizard should guide users through defining each of these 7 stages, but make stages 3–5 the focus — that is where customer differentiation lives. Stages 1–2 and 6–7 can use smart defaults and templates.

1.3 Branching Patterns

Four types of branching appear across the analyzed scripts. These represent the range of complexity the wizard must support:

PatternDescriptionExamples
Binary gates Yes/no knockout questions that immediately qualify or disqualify Honeycomb's "do you have coverage?", Amanda's user count threshold
Conditional paths Different scripts triggered based on a categorical answer Apple DR's attorney/self-settled/bankruptcy paths, LavieLabs' skin type responses
Multi-scenario entry Different opening flows based on how the call connects UBS's 3 scenarios (direct / extension / gatekeeper), Dark Horse's service routing
Exploratory vs. directed Flow adapts based on how specific the prospect's answers are Amanda's Explorer vs Sprinter path based on answer specificity

Design Insight: The wizard should support conditional branching at each question, but default to simple linear flows. Advanced users can layer in gates and multi-path logic. The flow builder should make linear easy and branching possible, not the reverse.

1.6 Transfer & Handoff Patterns

Three outcome types emerge, each with distinct data requirements:

Hot Transfer

Real-time connect to a human rep.

  • Transfer brief template
  • Availability check logic
  • Fallback: if unavailable → schedule
  • Used by: LavieLabs, Honeycomb, Apple DR, Warranty First

Schedule Meeting

Book a future meeting or callback.

  • Calendar integration (Zoho, Google, custom links)
  • Time slot offering logic
  • Email capture for invite
  • Used by: UBS, Dark Horse (both), Honeycomb (fallback)

Disqualify / Soft Letdown

When the prospect does not meet qualification criteria, a polite exit is required. Amanda has the most sophisticated soft letdown with a dedicated sub-agent. Other scripts simply end the call politely and optionally create a support ticket or "close file" entry.

Transfer Brief: Gold Standard (Honeycomb)

Name:              [Lead Name]
Property:          [# units / type] in [state]
Premium confirmed: [Amount]
Current insurance: [Yes/No]
Decision maker:   [Yes/No]
Timeline:          [Effective date]

Design Insight: The wizard should ask "What happens when a prospect qualifies?" and "What happens when they don't?" with configurable templates for each outcome. The transfer brief builder should allow customers to define structured fields that get passed to the human rep.

Step-by-Step Wizard Design

STEP 1 OF 8

"Who is your agent?" — Identity

This step establishes the agent's persona. Fields include agent name, company name, role, tone selection (4-option picker with preview), AI disclosure toggle, and language selection.

  • Agent name (text input)
  • Company name (text input)
  • Agent role/title (text input, optional)
  • Company description (textarea, 2-3 sentences)
  • Products/services (tag input)
  • Quick tone picker with 4 options + live preview card
  • AI disclosure toggle
  • Language selection (multi-select)

Design Recommendation: Show a live preview card of the agent persona as the user fills in fields. The card updates in real time to display: the agent's name, role, a sample greeting in the selected tone, and language badges. This gives immediate feedback and makes the process feel tangible.

STEP 2 OF 8

"What's the call about?" — Context & Opening

Defines the call type, purpose, and opening line. The opening line builder supports variable insertion from a toolbar.

  • Call type selector (4 options with icons: Inbound / Outbound Cold / Outbound Warm / Outbound Confirmation)
  • Call purpose textarea
  • Opening line builder with {variable} insertion toolbar
  • Recorded line disclosure toggle + custom text

Design Recommendation: Render the opening line as a speech bubble preview below the input. As users type and insert variables, the bubble updates live. Show the call type icons as large, clickable cards (similar to a survey question layout) rather than a dropdown.

STEP 3 OF 8

"What data do you have?" — Pre-Call Parameters

Maps CRM fields to agent variables. Pre-populate common fields (prospect name, email, phone, company) and allow custom additions.

  • Variable list: name + type (string/number/date/boolean) + CRM source field
  • Pre-populated defaults: prospect_name, email, phone, company
  • "Add custom variable" button
  • Optional: CRM integration selector (Salesforce, HubSpot, Zoho, monday.com, custom API)

Design Recommendation: Use a visual CRM field mapping interface with an add-row pattern. Each row shows: variable name (editable), type selector, CRM source (editable). Consider a drag-and-drop interface for reordering. If CRM integration is connected, auto-suggest available fields.

STEP 4 OF 8

"What do you need to learn?" — Call Flow Builder

The core of the wizard. A sequential question builder where each question is a card with configurable properties. This is the step that distinguishes a simple LavieLabs-style script from a complex Amanda-style decision tree.

  • Question cards (sequential, draggable for reorder)
  • Each card: question text + answer type (yes/no, free text, multiple choice, numeric) + required flag
  • Conditional branch: "If answer is ___, then ___" (proceed / skip to step / knockout)
  • Knockout/disqualify toggle per question
  • Section headers (Qualification, Discovery, Pitch, etc.)
  • CRM field mapping per question (where to save the answer)
  • Internal notes field (not sent to prospect)

Design Recommendation: Build a visual flow builder — linear by default with connected cards, similar to Typeform's builder but with conversation semantics. Users see the flow as a vertical chain of cards. To add a branch, click "Add condition" on any card and a fork appears. The interface should make linear easy and branching possible (not the reverse). Include section dividers that users can name (e.g., "Qualification," "Discovery").

STEP 5 OF 8

"How to handle pushback?" — Objection Handling

Paired objection-response cards. Each pair defines what the prospect might say and how the agent should respond.

  • Paired cards: Objection (left column) → Response (right column)
  • Optional "Goal" field: what the response aims to achieve
  • Optional "Then" action: return to flow / transfer / schedule
  • "Add objection" button at bottom
  • "Auto-generate from company info" AI button
  • Pre-populated templates by industry

Design Recommendation: Use a two-column card layout. Left side shows the objection in a red-tinted card, right side shows the response in a green-tinted card. Include an "Auto-generate" AI button that analyzes the company description and call purpose to suggest common objections for that industry. Start with 3 pre-filled examples and allow adding more.

STEP 6 OF 8

"What can your agent answer?" — FAQs & Knowledge

Knowledge base for the agent. Includes both things the agent can answer and things it explicitly cannot discuss.

  • Q&A pair list (collapsible accordion)
  • "Topics to avoid" list with fallback behavior
  • Fallback response template: "I will transfer you to someone who can answer that"
  • Optional: upload company docs / paste website URL for auto-extraction

Design Recommendation: Use a collapsible accordion for each Q&A pair. Include a "Paste your website URL" field with an AI extraction button that crawls the page and suggests FAQ pairs. The "Topics to avoid" section should use red-tinted list items with a default fallback message that users can customize.

STEP 7 OF 8

"What happens next?" — Outcomes & Routing

Defines the two primary paths: what happens when a prospect qualifies, and what happens when they do not.

  • Qualified path: Hot transfer / Schedule meeting / Send info
  • Transfer brief template builder (structured field list)
  • Calendar integration details (link, provider)
  • Follow-up email/link configuration
  • Unqualified path: Soft letdown script / Raise ticket / Close file
  • Soft letdown script textarea
  • Hold/waiting behavior: conversation instructions while on hold

Design Recommendation: Use a two-lane layout: left lane (green border) for the qualified path, right lane (neutral border) for the unqualified path. Each lane has its own outcome type selector and configuration area. The transfer brief template builder should let users add fields with name + type, and show a preview of the brief as structured data.

STEP 8 OF 8

"Any rules?" — Compliance & Guardrails

The final step defines hard constraints: things the agent must never do, things it must always do, and behavioral settings for edge cases.

  • "Never say" list (red-coded items)
  • "Always say" list (green-coded items)
  • Silence timeout: seconds before retry (default 5s)
  • Silence retry behavior: repeat question / prompt / end call
  • Maximum call duration (minutes)

Design Recommendation: Use simple list builders with color coding: red background for "never" items, green background for "always" items. Include pre-populated suggestions based on industry and call type (e.g., financial services automatically gets "do not promise specific returns" and "do not provide legal advice"). Silence and duration settings should use sliders with sensible defaults.

Design Recommendations Summary

  • Progress stepper at the top: 8 clearly labeled steps with completion indicators. Users can jump between steps after initial completion.
  • One screen per step: Each step should fit on one screen without scrolling marathons. If content overflows, use collapsible sections within the step.
  • Preview Agent button: Available from any step. Opens a side panel that shows a simulated conversation based on current configuration.
  • "Paste existing script" option: On Step 1, offer an AI-powered import that parses a pasted script and auto-fills all wizard steps. This dramatically reduces onboarding friction.
  • Industry templates: Pre-built configurations for Insurance, SaaS, Financial Services, Auto, Professional Services, and D2C/Retail. Each template pre-fills all 8 steps with sensible defaults.
  • Mobile-responsive: Designed for quick edits on mobile. The full builder experience is desktop-first, but individual field edits work on smaller screens.
  • Script versioning: Support V1/V2 versioning with diff view. Users should be able to duplicate a script, modify it, and A/B test (as UBS does with 4 scheduling options).
  • Multi-script per customer: Support multiple scripts for different funnel stages (as Dark Horse uses Scheduler + Validator for the same company).

Key Design Decisions for the Prompt

1. Sectioned Architecture

Each section maps to a wizard step. This makes it easy to regenerate individual sections when the customer updates their configuration without rebuilding the entire prompt. A change to objection handling only regenerates the OBJECTION HANDLING section.

2. Stage-Based Flow

Rather than one giant instruction block, the call is broken into numbered stages. This gives the LLM clear sequential guidance and aligns with how voice AI platforms (like OneAI) structure their script-building flows as goal-based nodes.

3. Separation of Concerns

Four distinct layers with different change frequencies:

  • Identity/Tone (who the agent IS) — changes rarely, maybe once at setup
  • Call Flow (what to DO) — changes per campaign or A/B test
  • Objections/FAQs (what to KNOW) — grows incrementally over time
  • Guardrails (what NOT to do) — set once, enforced always

4. Variable Injection

All dynamic data uses {curly_brace} syntax, injected at call time from the CRM. The prompt template itself is static; the variables are dynamic. This matches the convention every analyzed script already uses naturally.

5. Extracted Data Schema

The prompt defines upfront what structured data the agent should extract and return at the end of each call. This enables automated CRM write-back and ensures consistent data capture across all calls.

6. Conditional Logic in Flow

Simple if/then rules within each question rather than complex decision tree notation. The LLM handles nuance and natural conversation flow; the structured rules handle routing decisions. This keeps the prompt readable while supporting the branching patterns identified in Section 1.3.

Prompt Size Considerations

Complexity LevelEquivalent PlaybookEstimated TokensCharacteristics
Simple LavieLabs ~500 tokens Linear flow, 3-5 questions, no objections, single outcome path
Medium Honeycomb, Warranty First ~1,500 tokens 5-10 questions, 3-5 objections, binary gates, transfer brief
Complex Amanda, Dark Horse, UBS ~3,000+ tokens Decision trees, 10+ questions, 10+ objections/FAQs, multi-path routing

Token Optimization Recommendation

Objection handling and FAQs are the biggest token consumers. For complex scripts exceeding 2,000 tokens in the core flow:

  • Keep the core flow (Identity + Tone + Call Context + Call Flow + Guardrails) in the system prompt — target under 2,000 tokens
  • Load objections and FAQs as separate retrievable context via tool/function calls, or inject them as supplementary context only when the prospect raises a question or objection
  • This reduces baseline token cost per call while maintaining full coverage when needed
